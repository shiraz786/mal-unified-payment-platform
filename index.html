Unified Payment Data Pipeline

Senior Data Engineer Assessment submission for Mal — cross-product canonical payment model, validation, contract versioning, and migration strategy.

Cards Transfers Bill Payments Canonical Schema Validation v1 → v2 Migration

Problem

Three product squads produced payment events with different field names, structures, and rules. This project standardizes them into one reusable payment model for downstream analytics, reconciliation, risk monitoring, and product reporting.

Input Sources

  • Cards CSV
  • Transfers CSV
  • Bill Payments CSV

Processing

  • Field standardization
  • Validation checks
  • Error capture

Outputs

Canonical Payment Schema

Field Meaning
event_idUnique payment event identifier
payment_typecard / transfer / bill_payment
customer_idCustomer identifier
amountTransaction amount
currencyISO currency code
event_timestampStandardized event timestamp
statusNormalized status
payment_methodRail or method used
source_systemOriginating squad/system
contract_versionData contract version

Validation Rules

Checks

  • Mandatory fields present
  • Amount is positive
  • Currency format valid
  • Timestamp parsable
  • Status normalized

Error Handling

  • Invalid rows are not dropped silently
  • Errors written to validation_errors.jsonl
  • Run summary captures valid vs invalid counts

Contract Versioning

The pipeline includes a simple data contract evolution example from v1 to v2, showing how canonical entities can change in a controlled way while keeping backward migration clear.
